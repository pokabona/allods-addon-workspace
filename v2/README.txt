Документация Allods для локальной разработки аддонов

Корень проекта
--------------
Рабочая папка проекта может находиться на разных ПК под разными буквами
Google Drive, например:

G:\Мой диск\Аллоды проги
H:\Мой диск\Аллоды проги

В этой документации такой путь обозначается как <AllodsWorkspace>. Скрипты v2
должны вычислять корень относительно своей папки, а не полагаться на букву
диска. Для явной проверки текущего расположения можно выполнить:

.\resolve-allods-workspace.ps1

Назначение папки v2
-------------------
Папка v2 содержит структурированную JSON-базу знаний для быстрого поиска,
анализа API и работы ИИ.

Файлы:

1. allods_manifest_index.json
   Компактный навигационный индекс. Помогает определить, в каком файле
   искать нужную информацию. Повторяющиеся пути, категории и объявления
   намеренно не дублируются в каждой записи.

2. allods_search_index.jsonl
   Служебный полнотекстовый индекс для search-v2.ps1. Одна JSON-запись
   на строку. Автоматически пересобирается и не является источником истины.
   Не прикладывать этот файл целиком к запросам ИИ.

3. allods_api_core.json
   Текущий API, события, функции, типы, виджеты и основные правила.
   Это главный источник для проверки существования и объявления API.
   Содержит только правила использования документации API.

4. allods_examples_samples.json
   Примеры Lua/XDB и содержимое демонстрационных аддонов.
   Примеры помогают с реализацией, но не заменяют текущий API.
   Поиск охватывает как examples, так и вложенные samples.

5. allods_runtime_knowledge.json
   Проверенные в игре сведения, известные формы и дерево виджетов,
   результаты экспериментов, рабочие и нерабочие подходы.
   Runtime-правила хранятся здесь, а не в allods_api_core.json.

6. allods_history_changelog.json
   История изменений, удалённые и устаревшие элементы.
   Не считать элемент текущим API, если его нет в allods_api_core.json.

7. allods_addons_knowledge.json
   Снимок текущих PAK-файлов из соседней папки "адоны".
   Содержит состав пакетов, исходный Lua/XDB/текст, хеши и метаданные
   бинарных ресурсов, а также связь с уже записанными runtime-отчётами.
   Наличие упоминания не означает, что точная версия проверена в игре.

Связь с ModdingDocuments
------------------------
Соседняя папка:

<AllodsWorkspace>\ModdingDocuments

должна храниться отдельно. Она содержит исходную структуру документации,
полные Lua/XDB-файлы, UniSlot, текстуры, бинарные ресурсы, готовые примеры
и LuaCompiler\luajit.exe.

JSON-файлы v2 удобны для поиска и понимания документации, но не заменяют
ModdingDocuments при разработке, проверке Lua, сборке аддонов и копировании
ресурсов.

Рекомендуемый порядок работы
----------------------------
1. Искать API и события в allods_api_core.json.
2. Смотреть реализацию в allods_examples_samples.json.
3. Проверять игровые особенности в allods_runtime_knowledge.json.
4. Смотреть текущую реализацию наших PAK в allods_addons_knowledge.json.
5. Использовать файлы из ModdingDocuments как исходники и шаблоны.
6. Обращаться к истории только для анализа старого или удалённого API.

Исходники и сборка
------------------
Исходники редактируемых аддонов должны лежать в:

<AllodsWorkspace>\projects\<AddonName>

Готовые текущие PAK хранятся в:

<AllodsWorkspace>\адоны

Сборка PAK без установки в игру:

.\build-addon-pak.ps1 -ProjectPath ..\projects\AutoVendorHelper_v1

Полный рабочий цикл для проекта: собрать PAK в папку "адоны", обновить v2 и
проверить свежесть записей:

.\build-addon-workflow.ps1 -ProjectPath ..\projects\AutoVendorHelper_v1

Установка в игру выполняется только явно:

.\install-addon-pak.ps1 -PakPath ..\адоны\AutoVendorHelper_v1.pak

По умолчанию install-addon-pak.ps1 берёт путь игры из переменной окружения
ALLODS_GAME_ADDONS_PATH. Если переменная не задана, используется запасной путь
C:\VK Play\Аллоды Онлайн\data\Mods\Addons.

Связь с папкой "адоны"
----------------------
Соседняя папка:

<AllodsWorkspace>\адоны

содержит реальные PAK-файлы и остаётся главным источником их точного
архивного состава и бинарных ресурсов. allods_addons_knowledge.json
предназначен для поиска по их коду и инвентаризации.

Это рабочая папка проекта, а не папка, из которой игра загружает аддоны.
После сборки новой версии сначала сохранить PAK сюда с новым номером версии,
не перезаписывая предыдущую рабочую версию без явной необходимости.

После добавления или замены PAK-файлов базу allods_addons_knowledge.json
нужно обновить. Проверенные в игре результаты и нерабочие подходы следует
отдельно записывать в allods_runtime_knowledge.json.

Установка и проверка аддона в игре
----------------------------------
Текущая установка игры:

C:\VK Play\Аллоды Онлайн

Готовый PAK для запуска в игре нужно копировать в:

C:\VK Play\Аллоды Онлайн\data\Mods\Addons

Внутри PAK ожидается структура:

Mods/Addons/<СистемноеИмяАддона>/AddonDesc.(UIAddon).xdb

Системное имя в <Name> и имя внутренней папки следует сохранять постоянными
между версиями. Номер версии указывать в имени PAK и отображаемом name.txt.
Смена системного имени создаёт отдельную папку Configs и пустой конфиг.

Перед проверкой убедиться, что:

1. В игровую папку скопирована именно новая версия PAK.
2. Старый PAK с тем же системным именем не создаёт конфликт.
3. Игра полностью перезапущена или аддоны перезагружены доступным способом.
4. Дата и размер PAK в игровой папке соответствуют собранному файлу.

Все сообщения, которые аддон отправляет в игровой чат, должны начинаться
с короткой подписи источника, например:

[AutoRez] сообщение
[LabMap] сообщение

Технический вывод в mods.txt также должен передавать системное имя аддона
первым аргументом common.LogInfo, если используется собственная обёртка.

Рабочая папка <AllodsWorkspace>\адоны используется для хранения,
анализа и индексации. Само наличие PAK в ней не устанавливает аддон в игру.

Логи и игровые конфиги
----------------------
Основной лог Lua-аддонов:

C:\VK Play\Аллоды Онлайн\Personal\Logs\mods.txt

При диагностике ошибки сначала брать свежий файл mods.txt после запуска игры
и воспроизведения проблемы. Проверять время изменения файла, чтобы не
анализировать лог от предыдущего запуска.

Основные пользовательские конфиги игры:

C:\VK Play\Аллоды Онлайн\Personal\user.cfg
C:\VK Play\Аллоды Онлайн\Personal\user.cfg.bak
C:\VK Play\Аллоды Онлайн\Personal\global.cfg

Глобальные секции userMods, например LM_maps, сохраняются в Personal\user.cfg.
Перед ручным изменением user.cfg закрыть игру и сделать резервную копию.

Персональные конфиги некоторых аддонов находятся в:

C:\VK Play\Аллоды Онлайн\data\Mods\Configs\<ИмяАддона>\user.cfg

Не путать игровые конфиги с копиями из Downloads, Temp, backups или папок
проекта. Такие копии полезны для анализа и восстановления, но игра их
не читает.

Локальные инструменты
---------------------
search-v2.ps1 выполняет короткий поиск без загрузки всей документации.
Обычный поиск читает allods_search_index.jsonl построчно и не разбирает
полные JSON-базы. Исходный файл загружается только с -Full или -Raw.

Обычный поиск текущего API через компактный manifest:

.\search-v2.ps1 "GetTargetSales"

Получение компактной полной записи из API:

.\search-v2.ps1 "GetTargetSales" -Full

Получение исходной записи со всеми служебными полями (редко нужно):

.\search-v2.ps1 "GetTargetSales" -Raw

Поиск по исходникам наших PAK:

.\search-v2.ps1 "AutoRez" -Scope Addons

Поиск по проверенным runtime-отчётам:

.\search-v2.ps1 "bank-columns" -Scope Runtime

Доступные области поиска:

Api, Runtime, Examples, Addons, History, All

Для машинной обработки можно добавить параметр -Json. Количество результатов
задаётся параметром -MaxMatches; по умолчанию возвращается не более 5 записей.

Не прикладывать JSON-файлы целиком к запросу ИИ. Сначала использовать поиск,
а затем передавать только найденные записи. Обычный поиск экономнее -Full,
а -Raw следует применять только для диагностики структуры базы.

После изменения папки "адоны" выполнить:

.\update-addons-knowledge.ps1

Скрипт заново создаст allods_addons_knowledge.json и обновит сведения
о пакетах в allods_manifest_index.json. Исходные PAK-файлы не изменяются.

Перед работой с пакетами или после ручных операций можно проверить свежесть
снимка:

.\check-addons-freshness.ps1

Проверка сравнивает PAK-файлы из папки "адоны" с allods_addons_knowledge.json
и allods_manifest_index.json по именам, размерам, времени изменения и sha256.
Сгенерированные пути к PAK в JSON используют логический префикс
<AllodsWorkspace>, чтобы записи были переносимы между дисками G: и H:.

Для безопасного аудита структуры без удаления и конвертации файлов выполнить:

.\audit-allods-workspace.ps1

Скрипт выводит группы полных дубликатов, пустые файлы, абсолютные пути,
кандидатов Windows-1251 и отфильтрованные TODO/FIXME/BUG. Для проверки generated JSON добавить -IncludeGeneratedKnowledge. Для полного
машиночитаемого отчёта использовать параметр -Json.

Для внешнего ревью или локального анализа создать чистый пакет:

.\export-review-pack.ps1 -OutputPath <куда-собрать>

Пакет исключает _archive, ModdingDocuments, тяжёлый search-index и живые
игровые папки. Передавать на ревью следует содержимое этого пакета, а не весь
Google Drive workspace.

После замены API- или runtime-документации выполнить:

.\update-manifest-index.ps1

Скрипт автоматически запустит optimize-api-core.ps1, удалит из API-текстов
сгенерированные навигационные хвосты Search/Связанные страницы, разнесёт
API- и runtime-правила, пересоберёт manifest и полнотекстовый search index.

Резервные копии
---------------
Резервные копии v2 хранятся в _archive\v2-backups и не участвуют в поиске.
Рабочая корневая папка не должна содержать файлы *.bak.

Не переносить содержимое ModdingDocuments в v2 и не удалять эту папку.
